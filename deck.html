The Electronic Voting Paradox

Why Existing Systems Fail

⚡ Centralized SQL
A single UPDATE query by a rogue DBA can silently alter election outcomes. Server logs are trivially purged. Zero cryptographic protection.
💸 Public Blockchains
Ethereum tops out at ~15 TPS. Gas fees spike to $50+ per vote. Metadata leakage via transaction graphs de-anonymizes voters through forensic analysis.
🔒 Hardware E-Voting
Impenetrable remotely but vulnerable to supply-chain attacks, pre-loaded malicious firmware, and Rowhammer memory exploits. Audits require physical access across geographies.
📜 Paper Ballots
Susceptible to ballot stuffing, missing ballot boxes, and local coercion. Manual recounts take weeks and are plagued by human error. Non-cryptographic by nature.

Database Design Theory

Normalization & ER Design

1NF ✓
All attributes are atomic. No multi-valued columns. Each row represents a single tuple with a unique UUID primary key.
2NF ✓
Single-attribute PKs eliminate partial dependencies entirely. All non-prime attributes are fully functionally dependent on PK.
3NF + BCNF ✓
No transitive dependencies. For every FD X→Y, X is a superkey. Party depends only on Candidate(id), not on name.
Relational Schema
Users(id, name, email, voterHash)
Election(id, title, status, merkleRoot)
Candidate(id, name, party, electionId)
   FK: electionId → Election(id)
Vote(id, voterHash, electionId, candidateId,
    voteHash, prevHash, timestamp)
   FK: electionId → Election(id)
   FK: candidateId → Candidate(id)
   UNIQUE(voterHash, electionId)
